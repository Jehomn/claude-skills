## 一、核心方法论：五阶段内容管道

这不是流水线，是一个逐层把关的质量体系。每阶段的产出是下一阶段的输入，不可跳跃。

```
阶段1: 选书 → 阶段2: 全量索引 → 阶段3: 精读对比 → 阶段4: 内容稿 → 阶段5: PPT生成
```

### 阶段 1：选书（1-2h）

**目的：** 在 27 本 PDF 中确定哪些值得参与。

**做法：**
- 书名含 Anesthesia / Analgesia / Pain / 麻醉 / 镇痛 → Tier 1（全模块参与）
- 专科书（监测/心电图/急救/字典）→ Tier 2（按模块按需调用）
- 中文教材额外纳入（受众是中文执业兽医，术语对照不可缺）

**产出：** Tier 1/2 分类表，附每本书概览摘要。

**踩坑：** 不先做这一步，阶段 2 会对 27×4=108 个组合做索引，一半以上是无效工作。

---

### 阶段 2：全量索引（3-5h）

**目的：** 精确定位每本书对每个模块的覆盖程度。

**做法：**
- 用 `pdftotext` 提取 PDF 文本，按关键词检索
- 人工判定相关性（机器不区分麻醉学术语的同义词/缩写）
- 索引粒度：章节号 + 页码范围 + 知识点清单（用作者原术语）
- 相关性评分：3=专章论述（≥5页连续） / 2=部分涉及（1-4页） / 1=间接相关 / 0=不相关

**产出：** 每模块的 `index_m*.json`，含每本书的评分+知识点清单+定位。

**踩坑：** 评分不能全自动——搜索引擎分不清 "hypotension" 和 "MAP < 60 mmHg" 是同一个东西。

---

### 阶段 3：精读对比（4-8h/模块）

**目的：** 从高分书中提取具体数据，标注教材间差异。

**做法：**
- `pdftotext -f <start> -l <end>` 提取高分章节原文
- 按知识点整理对比表（剂量/阈值/机制/证据级别）
- 差异用 `[差异]` 标签标注，并列双方观点+各自依据，不下无证据的判断

**产出：** 差异报告（`m*_extracts/` 目录下的 .txt/.md 文件）

**核心原则：** 两本书结论冲突时，只列证据、不做裁判。你不知道哪个观点在具体病例中更适用。

---

### 阶段 4：内容稿写作（每模块 3-6h）

**目的：** 把原始教材信息转化为教学用的幻灯片内容——重组知识拓扑、提炼因果链、标注出处。

**格式规范：**

```markdown
### P2.2.2 | α₂ 激动剂药物对比
| 模块: M2 | 节: 2.2 | 来源: Lumb & Jones 6th Ch.22, Pharmacol Vet Anaesth Ch.4

[正文内容，含出处角标 [1][2]]

[差异]（如有教材间冲突）

[参考文献]
1 Lumb & Jones 6th Ed., Ch.22 Table 22.1
2 Pharmacol Vet Anaesth, Ch.4 Table 4.1
```

**写作铁律：**
- 每条结论有出处角标，无出处不写
- 数字必须具体（「MAP < 60 mmHg 持续 > 3min」而非「血压过低」）
- 机制用因果链：「A → B → C」
- 禁止句式：「值得注意的是」「大量研究表明」「在临床实践中」「对于……来说」
- 所有临床实例必须来自教材真实数据，禁止凭记忆/常识给数值
- 每页引用 ≤ 4-5 条，完整格式不缩写
- 中文引号用「」，不用 ASCII "

**为什么内容稿必须独立存在：**
- 审内容效率远高于审 PPTX（Markdown 可滚动阅读+全局搜索 vs PPTX 逐页翻）
- 内容资产与输出格式解耦——将来换生成引擎，内容稿不用改
- 内容稿是阶段 4 的「验收文档」——确认后再写代码

---

### 阶段 5：PPT 生成（每模块 2-4h 编写 + 迭代）

**为什么走了弯路又回来：** 最初尝试 HTML→PPTX 三条路线：

| 路线 | 失败原因 |
|------|---------|
| dom-to-pptx | CSS flexbox 坐标提取不可靠，多文件 `<style>` 拼接导致级联污染 |
| Playwright 截图 | Headless Chrome 渲染与桌面 Chrome 不一致，字体度量有差异 |
| html2canvas 桥接 | 截图方案产生 26MB+ 文件，字体不可编辑 |

**最终方案：** pptxgenjs 直接编码。跳过一切中间翻译层，直接在 PPT 坐标空间中生成。

**核心教训：** 任何「先渲染再翻译」的方案，在字体度量、CSS 级联、跨引擎渲染差异面前都会崩溃。在 PPT 引擎中原生生成是唯一可靠的路径。

## 二、代码架构：模板-数据分离

这是 167 页能用 ~1500 行 JS 完成的核心原因。

### 模板体系（6 种，只写一次）

| 模板 | 用途 | 核心参数 |
|------|------|---------|
| `contentSlide` | 正文+提示 | `sec, title, sub, paras[], tip, refs[]` |
| `tableSlide` | 数据表格 | `sec, title, intro, tbl{h, r, cr}` |
| `cover()` | 封面 | 讲师信息 + 四模块目录 |
| `intro()` | 个人介绍 | 照片 + 分类信息 |
| `modTitle()` | 模块标题页 | 节列表（自动适配 M1/M2/M3/M4） |
| `refBooks()` | 参考书目 | 分组双栏 → 后来改为双栏独立组 |

plus 专用页：`m1s1Summary()`（卡片式小结）、`capnoPlaceholderSlide()`（留白贴图）

### 数据驱动模式

每一页是一个 JS 对象，灌入模板：

```javascript
{ t:'table', mi:1, pim:7, sec:'2.2 镇静药', title:'α₂ 激动剂药物对比',
  intro:'...', tbl:{h:[...], cr:[...], r:[[...]]}, refs:[...], pn:0 }
```

`t` 选择模板类型，`mi` 选择模块色，`pim` 计算进度条。`pn` 由 main() 的全局计数器自动赋值——删掉了所有手写死页码。

### 模块合并

M1 和 M2 最初独立文件，后来合并为连续讲课用。关键设计让合并几乎零成本：
- 模板函数和数据数组完全解耦
- `modTitle()` 接受 `modKey` 参数自动切换模块色和节列表
- `addProgressBar()` 接受 `modIdx` 参数控制哪个模块是当前
- 合并只是 `main()` 里加了几个 `add()` 调用

## 四、进度条系统

一个被反复打磨的子系统，值得单独记录。

### 状态机

```
已完成模块：纯色填充 + 白字
当前模块：  左侧纯色（比例=pageInMod/pages）+ 右侧浅灰 + 深色字（<45%）/ 白字（≥45%）
未开始模块：浅灰 + 淡字
```

### 技术要点

- `shrinkText: true, wrap: false` —— 窄段文字自动缩小，绝不换行
- 当前模块文字颜色在进度 ≤45% 时用深色（`#1C2E40`），>45% 时自动切换白色——因为文字居中，进度到 45% 时开始接触文字左边缘
- 段间 0.02" 透明间隙，视觉呼吸感
- 底部 0.008" 浅色分割线

### 跨模块过渡

合并文件中 `addProgressBar(slide, modIdx, pim)` —— `modIdx` 控制哪个模块是当前。M1 阶段 M1 是 current，M2 阶段 M1 complete + M2 current。自动过渡，无手动干预。

## 五、页码系统

### 规则

- 封面、个人介绍、参考书目、各模块标题页：无页码
- 正文从 M1 第一节开始计 01，跨模块连续计数
- 文本框宽 0.5"，两位数不换行；`padStart(2,'0')` 保持两位数格式

### 实现

`main()` 内全局计数器 `p`，每调用一次 `add()` 自增并覆写 `data.pn`。专用页（summary / capno）手动 `++p`。

### 踩坑

最初每个 slide 函数硬编码 pageNum，后来手动维护时出现重复编号（79 出现两次）。改为计数器后彻底解决。

## 六、中文排版铁律

### 引号

**必须用「」，禁止 ASCII "。** 违者 JS 字符串解析直接报错。

### 防换行（anti-widow）

- 进度条：`shrinkText + wrap: false`
- 模块标题页编号 "M1"：文本框 ≥ 2.5"（52pt 下约 1.5" 宽，留足余量）
- 正文：最宽可用区域（SW - 2*GX = 12.3"），16pt 中文每行容纳 ~68 字
- 窄栏标签：回避左右并排布局，改用标签在上/内容在下的纵向布局

### 间距计算

- 中文约 0.14"/字（10pt）、0.22"/字（16pt）
- 英文约 0.06"/字（10pt）
- 每次新增文本→心算字符数×字号对应的宽度→对比可用宽度→确认不换行

## 七、留白控制

用户要求全局留白 ≤20%。

### 计算方法

```
内容占用率 = (内容底部 - 内容顶部) / (refs分隔线 - header底部)
留白率 = 1 - 内容占用率
```

### 调节手段

| 参数 | 调整方向 | 效果 |
|------|---------|------|
| 表体字号 | ↑ | 每行更高 → 表格占更多空间 |
| 表体行高 | ↑ | 同上 |
| 正文字号 | ↑ | 更多行 → 更多空间 |
| 引导文字号 | ↑ | 同上 |
| 节标题/标题字号 | ↑ | header 区更高 |

### 典型值

- ASA 分级表（6 行）：13pt 表体 + 0.46" 行高 → 留白 ~16%
- 正文+临床提示页：16pt 正文 + 5 段 → 留白 ~18%

### 反面案例

3 行 bullet 正文页（最初 M1 小结的 v1 版本）→ 留白 ~50%。解决方案：改为 4 张编号卡片，块高 1.0"×4 → 留白降至 ~15%。

## 八、照片与图像处理

### 照片（个人介绍页）

进化路径：基础 `addImage` → 苹果风三层堆叠。

**最终方案：**
- 底层：M1 深蓝半透明卡片，`transparency: 88`，偏移右下 0.28"
- 中层：白色卡片，软阴影 `blur: 12, opacity: 0.10`，偏移 0.12"
- 顶层：照片 `sizing: { type: 'contain' }` + 圆角 0.08 + 深影 `blur: 14, opacity: 0.14`
- 背景色 `#F5F5F7`（Apple 暖浅灰）

**技术要点：**
- 照片比例 2:3（3840×5760），显示框 3.9×5.85" 严格匹配，不拉伸不裁剪
- `sizing: 'contain'` 保险——即使比例微小偏差也不变形

### 波形图尝试（失败记录）

用细矩形旋转拼曲线模拟 CO₂ 波形 → 效果差。**决策：改为空白占位页（虚线框+提示文字），留给用户手工贴图。** → 这是正确的止损——不要用代码强行模拟非代码擅长的东西。

## 九、引用与缩写系统

### 书名缩写

13 本书的标准缩写表写入 `CLAUDE.md` 作为全局唯一真相源。

**缩写原则：**
1. 含编者/书名关键信息，不能只剩姓氏（禁止「Lumb」）
2. 版次始终跟随
3. 中英文各自独立表述，不混用

**案例：**
- `Lumb & Jones 6th` ← Veterinary Anesthesia and Analgesia, 6th Ed. (2024)
- `Vet Anaesth Principles 2nd` ← Veterinary Anaesthesia Principles to Practice, 2nd Ed. (2020)
- `Q&A Small Anim Anesth` ← Questions and Answers in Small Animal Anesthesia (2016)

### 引用系统

- 正文内 `[1][2]` 标记（上标渲染）
- 页面底部参考文献行：`[1] 完整书名, 章节 pp.起-止`
- refs 分隔线在 `refsY - 0.08`，所有内容元素底部必须在分隔线上 ≥ 0.04"

## 十、开发流程模式

### 样本-扩展模式

1. 先做 4 页样本验证模板体系 → 确认配色/字体/间距/进度条全部跑通
2. 再批量填充剩余内容
3. 每模块独立开发，最后合并

**价值：** 避免「写完 167 页才发现排版有问题」的大返工。

### 迭代节奏

这次 M1 经历了 v1→v9 的迭代，覆盖了：
- 封面讲师信息、进度条配色、双栏渲染 bug、页码系统重构、换行修复、留白控制、波形图尝试→占位

**模式：** 每个版本解决 2-3 个具体问题，build 后立即打开 PowerPoint 验证。不要攒 10 个问题一口气改。

### 文件管理

- 开发中版本用递增后缀（v1/v2/v3...）
- 保留最近 2-3 版（往前两版及以上删掉）
- PowerPoint 锁定文件时换新版本号，不阻塞工作流
- 最终产出用语义化命名（`麻醉授课.pptx`）

### 模块合并时机

最初 M1/M2 独立文件。用户提出「要连续讲课」→ 合并。关键前提是模板-数据解耦架构已经就位，合并成本极低。

## 十一、常见陷阱速查

| 陷阱 | 原因 | 解决方案 |
|------|------|---------|
| HTML→PPTX 永远不对 | 浏览器和 PPT 排版引擎不互通 | 跳过翻译层，直接 pptxgenjs 编码 |
| 页码两位数换行 | 文本框太窄（0.3"） | 文本框 ≥ 0.5" |
| 进度条窄段文字换行 | 段宽不足（M1 20/160=1.66"） | `shrinkText + wrap: false` |
| 窄标签栏换行后剩 1-2 字 | 左右并排标签宽度不够 | 改用标签在上、内容在下的纵向布局 |
| 当前模块白字看不清 | 白字坐在浅灰底上（进度 0%） | 进度 ≤45% 用深色字，>45% 自动切白字 |
| 封面 M1 目录条不可见 | M1 色=背景色 | 封面目录条用浅色系 |
| 内容挤压页脚 | 扩大字号后未验证与 refs 间距 | refs 分隔线做排版安全边界，所有元素底部在线以上 |
| 页脚 refs 与页码重叠 | refs 高度 0.55" 到底 | refs 高度收至 0.45"，页码下移至 SH-0.25 |
| 表体留白 >30% | 字号太小+行高不够 | 表体 13pt + 行高 0.46"，引导文 16pt |
| JS 字符串解析失败 | 中文内容含 ASCII " | 全局使用「」 |
| `.mjs` 不能用 `require` | ESM 模块系统 | 用 `.js`（CJS） |
| pptxgenjs ShapeType 不可用 | v4 API 变更 | 用字符串 'rect' |

## 十二、资料库建立逻辑

这是这次最被低估的资产。

### 分层

- **Tier 1（核心圈）：** 书名含 Anesthesia/Analgesia/Pain/麻醉/镇痛 → 全模块参与
- **Tier 2（按需调用）：** 专科书 → 仅对应模块调用

### 索引

- 每本书×每模块的相关性评分（0-3）
- 精确到章节号+页码范围
- 知识点清单用作者原术语

### 证据提取

- `pdftotext` 按章节提取原文→人工标注关键数据
- 跨书对比→差异报告→内容稿的输入

### 关键教训

- 27 本不可全量处理，必须分层
- 中英文教材分开管理，但都纳入索引
- 索引是一次性投入（约 5h），但为后续三个模块节省了至少 20h 的重复搜索

## 十三、复盘：如果重来一次会怎么做

1. **一开始就做合并架构。** M1/M2 独立文件后再合并虽然成本低，但一开始就设计为合并可以省一次重构。

2. **波形图直接用占位页。** 尝试用代码画波形浪费了约 1h。代码画图形不是 pptxgenjs 的强项——直接留白让用户贴图是更好的分工。

3. **内容稿先于 PPTX 的铁律可以更早确立。** M2 一开始跳过了内容稿直接写 JS，被用户纠正。这个教训现在已经固化为方法论。

4. **封面目录一开始就该有。** 用户在第 9 轮才提出加目录，但其实第一版就应该有——四个模块的完整结构从项目一开始就是已知的。

5. **页码系统应该从第一天就设计好。** 硬编码页码导致后期大规模重构。如果一开始就用计数器方案，至少省 2h。

## 十四、深层模式：超越本项目的方法论

以下模式是领域无关的——适用于任何「从大量源材料生成教学PPT」的项目。

### 14.1 真相源层级（Source of Truth Hierarchy）

```
PDF原文 → 内容稿(Markdown) → JS数据 → PPTX
  ↑                          ↑        ↑
  不可篡改                   可审阅   可演示
```

**原则：** 永远只编辑最上游的源。发现 PPTX 有错 → 追溯到内容稿修正 → 重新生成。不要直接修 PPTX。这个层级结构保证了每一层都可以从上一层重建。

### 14.2 最小可行内容密度（Minimum Viable Content Density）

内容是自变量，排版是因变量。不要先定排版再塞内容——先定内容需要的空间，再调整排版来匹配。

**校准方法：** 做 3-4 页不同类型样本 → 测量每种类型的实际空间占用 → 反推字号/行高/间距 → 全局统一。样本-扩展模式（第十章）就是这一原则的操作化。

### 14.3 审阅模态分离（Review Modality Separation）

| 审什么 | 用什么工具 | 为什么 |
|--------|-----------|--------|
| 内容正确性 | Typora (Markdown) | 滚动阅读+全局搜索，比 PPTX 快 3 倍 |
| 排版视觉效果 | PowerPoint | 所见即所得 |
| 代码/数据结构 | 代码编辑器 | 语法高亮+搜索替换 |

**反模式：** 用 PPTX 审内容（逐页翻 109 页）、用 Markdown 审排版（看不到实际渲染）。

### 14.4 渐进复杂度展开（Progressive Complexity Disclosure）

```
样本页(4p) → 单模块(23p) → 合并(70p) → 全量(109p)
```

每个阶段验证一个假设：
- 样本：模板体系可行吗？
- 单模块：内容密度和排版协调吗？
- 合并：跨模块过渡顺畅吗？
- 全量：所有边界情况都覆盖了吗？

**原则：** 在每个复杂度层级上建立信心，再进入下一层。不要在样本阶段就试图解决全量问题。

### 14.5 约束先行设计（Constraint-First Design）

用户开出的约束清单：
- 纯μ激动剂不纳入麻前用药
- 犬猫限定（不涉及大动物/异宠）
- 所有医学数据从PDF查证，禁止凭记忆
- 中文引号「」
- 图表自生成，不截PDF图

**这些约束看似限制了自由，实际上极大简化了决策空间。** 当你知道什么不能做，剩下的选项就是清晰的。开新项目时，先列出约束清单——它比功能清单更重要。

### 14.6 人来判断美学（Human-in-the-Loop for Aesthetics）

代码能做的事情：布局、对齐、字体、间距、数据绑定、引用管理。

代码不能做的事情：判断「这页看起来舒服吗」「这个配色高级吗」「留白是不是太多了」「字体差异是不是太跳了」。

**模式：** 生成→展示→反馈→修改，闭环越短越好。这次每次改完 `node build + Start-Process` 约 30 秒，是 PPT 工程中可接受的反馈速度。

### 14.7 并行研究模式（Parallel Research Pattern）

M3 和 M4 的内容研究同时启动两个 agent，各自搜索不同模块的 PDF。两个模块共享同一套源材料（27本书），但研究任务完全独立。

**适用条件：** 多个子任务的源材料相同、输出格式相同、互相无依赖。**不适用条件：** 任务之间有顺序依赖（如必须先做 M1 才能做 M2）。

### 14.8 边界对象模式（Boundary Object Pattern）

CLAUDE.md 是人和 AI 共享的「边界对象」——两者都能读、都能更新。它承载了：
- 项目约定（目录结构、命名规则）
- 设计决策（配色、字体、引用格式）
- 当前状态（进度、待办）
- 缩写标准（13 本书的统一命名）

**原则：** 任何 AI 辅助项目中，都需要这样一个文件。它不替代代码文档（那是 README），不替代方法论文档（那是工程手册），它只做一件事：让人和 AI 在同一个上下文中工作。

### 14.9 逃生舱模式（Escape Hatch Pattern）

当自动化方案无法产生可接受的结果时，提供一个手工干预的入口。

**案例：** CO₂ 波形图——尝试用细矩形旋转拼曲线，效果差。决策：改为空白占位页（虚线框+提示文字），留给用户手工贴图。

**原则：** 识别「代码的边界」。有些东西（手绘波形、复杂示意图）用代码模拟的投入产出比太低。与其强求 80 分的自动生成，不如做一个干净的占位页 + 手工贴图——最终效果可能是 100 分。

### 14.10 决策日志模式（Decision Log Pattern）

每次结构性决策（放弃 HTML、改用卡片布局、重做页码系统）都在对话中留下了痕迹。这些决策事后被整理进方法论。

**原则：** 不要在事后「回忆」为什么做了某个选择——在选择的当下就记录。格式：「X 方案失败，原因 Y，改用 Z」。这些「为什么不这样做」的记录比「怎么做」更有长期参考价值。

### 14.11 单点故障意识（Single Point of Failure Awareness）

当前 `build_combined.js` 是唯一的构建入口，包含全部 109 页数据。如果它损坏 → 全部 PPTX 无法重建。

**缓解策略（未实施但应记录）：**
- 拆分为 `m1_data.js` / `m2_data.js` / `m3_data.js` / `m4_data.js` + `build.js`（组合器）
- 内容稿 Markdown 文件本身就是备份——即使 JS 全丢，可以从 Markdown 重建
- Git 版本控制提供时间维度的冗余

### 14.12 技能转移性（Skill Transferability）

这次积累的能力可以迁移到哪些场景：

| 能力 | 可迁移场景 |
|------|-----------|
| 模板-数据分离架构 | 任何 >50 页的结构化 PPT 项目 |
| 五阶段内容管道 | 任何从大量源材料生成教学内容的场景 |
| 进度条/页码系统 | 任何多章节连续演示 |
| 引用管理系统 | 任何需要标注出处的高密度学术 PPT |
| 苹果风设计令牌 | 任何需要高级感视觉的演示文稿 |
| 并行 agent 研究 | 任何多个独立模块同时开发的场景 |

不可迁移的：麻醉学专业知识、具体数值、药物剂量——这些是内容，不是方法。

### 14.13 速度与保真度的权衡（Speed-Fidelity Tradeoff）

| 方案 | 迭代速度 | 输出保真度 | 最终选择 |
|------|---------|-----------|---------|
| HTML→浏览器预览 | ~5 秒 | 高（浏览器渲染） | ❌ 无法翻译到 PPTX |
| HTML→dom-to-pptx | ~30 秒 | 低（CSS 污染） | ❌ 放弃 |
| Playwright 截图 | ~60 秒 | 中（字体差异） | ❌ 放弃 |
| pptxgenjs 直接编码 | ~30 秒 | 高（原生 PPT） | ✅ 采纳 |

**教训：** 在「看起来一样」和「实际是 PPT」之间，选后者。保真度优先于速度——错误的输出再快也没用。

## 十五、工程债务与改进方向

以下是在本次开发中识别到、但因时间/优先级未处理的事项。

### 15.1 构建脚本清理

当前目录残留多个开发中脚本（`build_samples.js`、`build_m1_full.js`、`build_m2_full.js`、`build_m2_samples.js`）。只应保留 `build_combined.js` 作为唯一构建入口。

### 15.2 数据文件拆分

`build_combined.js` 约 500 行，包含了模板函数 + 全部四个模块的数据。建议拆分为：
- `templates.js` — 模板函数
- `m1_data.js` ~ `m4_data.js` — 各模块数据
- `build.js` — 入口 + main()

降低单点故障风险，也方便单独重建某个模块。

### 15.3 Speaker Notes

pptxgenjs 支持 `slide.addNotes()`。109 页纯幻灯片信息密度极高。建议为每页自动附加关键数据摘要（剂量/阈值/参考文献），讲师备课和临场查阅会方便很多。

### 15.4 术语表

109 页散布大量英文缩写（MAC、ETCO₂、CRI、MMVD、BAOS、TIVA、PPV、GER、AKI……）。学员在模块三看到一个缩写可能忘了模块一的解释。在参考书目页后加入一页术语表会很有用。

### 15.5 模块间桥接语

模块间的逻辑衔接（「刚才讲了风险评估，现在进入用药策略」）目前只存在于讲师脑中。如果能以 speaker notes 或独立提示卡形式固化下来，这套课件就不仅是幻灯片，而是完整的课程脚本。

### 15.6 版本控制

整个项目不在 Git 下。`CLAUDE.md`、`PPT工程手册.md`、`build_combined.js`、`docs/*.md` 建议纳入版本控制。PDF 书库可选择性纳入或使用 Git LFS。

### 15.7 状态仪表盘

项目缺少一个一眼能看清四个模块各自在哪个阶段的看板。建议格式：Markdown 表格，四行（M1-M4），列：内容稿/PPT生成/最终确认，每格 ✅/⬜/⚠️。

## 十六、已挖更深：第二波挖掘

以下模式比前十四章更抽象——不是操作手册，而是元认知层面的洞察。

### 16.1 知识压缩比

27 本教材 → 109 页幻灯片。按每本平均 500 页计，约 13,500 页 → 109 帧。压缩比 ≈ 124:1。

**压缩方法论（隐含的，现在显性化）：**
- 删除：非犬猫内容、已淘汰药物、纯历史叙述
- 合并：多本教材对同一知识点的描述 → 一条综合陈述 + 差异标注
- 量化：所有模糊词（「可能」「通常」）替换为具体数字（「26×」「47%」「3h」）
- 结构化：因果链（A→B→C）而非列表（A, B, C）
- 分层：正文（必须知道）+ 临床提示（实操补充）+ 差异（学术诚实）

**原则：** 压缩不是删减——是重组。好的压缩让信息密度更高，不是信息量更少。

### 16.2 验证链

本项目隐含的验证链：AI 读 PDF → AI 写内容稿 → 用户（领域专家）审阅。每一步都可追溯：

```
PDF原文 → pdftotext提取 → AI精读 → AI写作 → 用户审阅 → 定稿
  ↑                                                            ↑
  可随时回溯验证                                             最终仲裁者
```

**这个链条成立的三个前提：**
1. 源材料可访问（PDF 未被移除/损坏）
2. 提取步骤可复现（pdftotext 命令行，非黑盒）
3. 最终仲裁者是领域专家（不是 AI、不是外行）

**模式：** 「AI 辅助专家写作」——AI 负责速度和广度（快速通读 27 本书、并行提取数据），专家负责深度和判断（识别错误、权衡矛盾、做临床判断）。

### 16.3 信任校准

用户对 AI 的信任在项目中呈现出明显的阶段性：

| 阶段 | 用户行为 | 信任水平 |
|------|---------|---------|
| 初期 | 反复要求「所有数据从 PDF 查证」 | 低——验证 AI 的事实准确性 |
| 中期 | 审阅 M1 内容稿，逐条确认 | 中——验证 AI 的方法论 |
| 后期 | M2/M3/M4 内容稿基本直接接受 | 高——信任已验证的方法论 |

**模式：** 信任的不是 AI 本身，而是「AI + 方法论」的组合。一旦方法论被验证（M1），后续产出（M2-M4）的信任建立就很快。

### 16.4 审美词汇共建

项目初期，用户和 AI 没有共享的审美语言。通过反复迭代，逐渐建立了一套共同词汇：

| 用户表述 | 翻译后的设计指令 |
|---------|----------------|
| 「高级感」 | 大量留白 + 软阴影 + 圆角 + 暖灰底色 |
| 「不跳脱」 | 字号差 ≤ 4pt，色差在同一色系内 |
| 「做满」 | 留白 ≤ 20%，通过字号/行高/间距调节 |
| 「苹果风」 | 暖浅灰底 + 三层堆叠 + blur 12-14 软影 + contain 保真 |
| 「干净」 | 去掉箭头符号、去掉不必要加粗、统一灰调文字 |
| 「抢戏」 | 元素超出其信息层级应有的视觉权重 |

**原则：** 不要等用户给出精确的设计参数——观察他们的评价词汇，翻译为设计决策。这个过程就是「审美对齐」。

### 16.5 双受众张力

PPT 同时服务于两个受众：

| 受众 | 需求 | 如何满足 |
|------|------|---------|
| 讲师（你） | 深度、证据、可追溯、备课参考 | 引用角标、参考文献行、完整剂量数据 |
| 学员（兽医） | 清晰、记忆点、临床可用 | 表格速查、临床提示框、小结卡、术语表 |

**未完全解决的张力：** Speaker notes 缺失意味着「讲师层」的信息目前嵌入在正文中（增加了学员的认知负担）。理想的架构是正文服务于学员，speaker notes 服务于讲师——但我们目前只实现了前半部分。

### 16.6 微观迭代的心理成本

每次 build + open 约 30 秒。项目全程约 50 次 build → ~25 分钟等待。

**观察到的行为适应：**
- 倾向于批量修改（3-5 个改动一次 build），而非逐个改动
- 批量越大，定位 bug 越难（不知道哪个改动引入了问题）
- 最优平衡点约 2-3 个改动/build

**工程启示：** 如果一个 PPT 生成系统的 build 时间 > 60 秒，开发者会倾向于更大的批次 → 更难调试。30 秒是一个可接受的心理阈值。

### 16.7 命名熵

项目文件命名经历了无规划的演化：

```
sample_output_v5.pptx         # 早期样本
M1_术前评估与风险把控.pptx     # 单模块独立
麻醉授课_M1+M2+M3_v3.pptx     # 合并但带版本号
麻醉授课_终版.pptx             # 语义化但不够精确
麻醉授课.pptx                  # 最终收敛
```

**问题：** 命名规则是事后建立的，不是事前设计的。

**可推广的命名约定：** `{项目}_{范围}_{版本}.pptx`，其中版本只在开发中追加，最终版不加版本号。`_终版` 这种名称应避免（「终版」之后通常还有「终版2」）。

### 16.8 范围蔓延的吸收

项目从「M1 样本」膨胀到「四个完整模块」的过程中，架构未发生断裂。原因：

- 模板-数据分离 → 加模块 = 加数据数组，不动模板
- 合并架构预埋 → 进度条支持 modIdx 参数，跨模块过渡是原生能力
- 内容稿独立 → 新模块来了先写 Markdown，确认后再写 JS

**反模式：** 如果当初每个模块用不同代码风格、不同模板实现，合并时会发生严重的集成痛苦。

**原则：** 为未知的增长预留接口（modIdx 参数、mi 字段、add() 辅助函数），但不过度设计。接口的成本是一次性的，重构的成本是累积的。

### 16.9 认知偏置：近因效应

长对话中，AI 受「上下文压缩」影响，对最近讨论的内容权重更高。这可能导致：
- 后期加入的功能（如术语表）被过度强调
- 早期解决的重大问题（如 HTML→PPTX 路线失败）被逐渐淡化

**缓解：** CLAUDE.md 和 memory 系统固定了关键信息，不依赖对话上下文。工程手册以结构化方式冻结了所有重要发现。

### 16.10 教学架构的隐性设计

四模块顺序不是任意的——它遵循临床麻醉的时间线：

```
M1 术前评估 → M2 用药策略 → M3 术中监测 → M4 术后镇痛
   ↑____________|_____________|_____________|
   评估风险 → 制定方案 → 执行+监控 → 收尾+过渡
```

**这是刻意为之的：** 每个模块是前一模块的自然延续。学员不需要「切换思维模式」——只需要在同一个临床故事线中前进。

**可推广模式：** 教学类 PPT 的模块排序应遵循受众的实际工作流，而非学科知识分类。按「做事的顺序」组织，不按「教科书的章节」组织。

### 16.11 视觉一致性与模块身份的平衡

```
统一元素：模板、字体、间距、引用格式、页码系统
差异元素：模块主色（M1深蓝/M2青蓝/M3蓝灰/M4灰蓝）
```

**这种设计让学员：**
- 看到颜色 → 立刻知道在哪个模块
- 看到布局 → 立刻知道这是什么类型的信息（表格/正文/小结）
- 不需要在每页重新学习「怎么看这页」

**原则：** 认知负荷分配给内容，不分配给格式。

### 16.12 引用密度的梯度设计

| 模块 | 引用密度 | 原因 |
|------|---------|------|
| M1 | 最高（每页 4-5 条） | 建立方法论可信度——「这个课是有证据的」 |
| M2 | 中高（每页 3-4 条） | 药物剂量需精确引用，但很多信息与 M4 重叠 |
| M3 | 中（每页 2-4 条） | 监测参数部分来自共识指南（AAHA），部分来自教材 |
| M4 | 中（每页 2-3 条） | 与 M2 共享药物数据，引用密度自然降低 |

**这是刻意为之：** 前段密集引用建立信任，后段信任已建立，引用服务于实质性争议点而非例行知识。

### 16.13 三语语境的认知开销

| 语境 | 语言 | 场景 |
|------|------|------|
| 医学内容 | 中文（主体）+ 英文（术语/缩写/药名） | PPT 正文 |
| 代码 | 英文（变量/函数）+ 中文（字符串内容） | build_combined.js |
| 人机交互 | 中文 | 对话 |

**未解决的问题：** 在 JS 代码中嵌入大量中文字符串（slide 标题、正文段落）——编码/换行/引号处理比纯英文代码更容易出错。这是「领域在中文但工具链是英文」的项目通病。

### 16.14 教学连续性的断裂点

109 页中，模块间的过渡只有一页标题页（深色底+节列表）。**讲师实际需要的过渡信息：**
- 「刚才我们讲了 X，现在进入 Y——它们之间的关系是 Z」
- 这些目前不存在于幻灯片上

**解决方案（未实施）：** 在每页 module title 的 speaker notes 中加入桥接语模板。例如 M2 标题页的 notes：「M1 告诉我们如何评估风险，M2 将告诉你如何用药来管理这些风险——两个模块的关系是『诊断→处方』。」

## 十七、第三波挖掘：如果再做一次会有什么不同

这次不是「如果重来」（第十三章已做），而是「如果把这个方法论移植到另一个领域会有什么通用挑战」。

### 17.1 领域适配的五个变量

| 变量 | 本项目取值 | 变化后的影响 |
|------|-----------|------------|
| 源材料格式 | PDF（可 pdftotext 提取） | 如果源是扫描图片/网页/视频 → 阶段 2-3 需要 OCR/转录/手工整理 |
| 内容确定性 | 高（教科书是定论） | 如果是快速演进的领域 → 需要版本标注+时效性管理 |
| 受众专业性 | 高（执业兽医） | 如果是公众/学生 → 深度降低+篇幅缩短+术语解释增加 |
| 讲师参与度 | 高（用户是领域专家+讲师） | 如果是为他人制作 → 需要额外的需求访谈环节 |
| 单次规模 | 109 页 | 如果是 500+ 页 → 需要更强的模块化和自动化 |

### 17.2 可复用的不是代码，是架构模式

模板函数（contentSlide/tableSlide）是领域相关的——它们的参数结构是为「医学教学 PPT」定制的。**可复用的：**
- 五阶段管道逻辑（任何「源材料→教学内容」项目）
- 模板-数据分离思想（任何「批量生成标准化输出」项目）
- 审美对齐方法（任何「AI 生成视觉内容」项目）
- 约束先行策略（任何「有明确用户偏好的创意项目」）
- 文件管理约定（任何「多版本迭代」项目）

### 17.3 本项目的独特优势（在其他项目可能不存在）

- 用户是领域专家——内容审阅不需要外部咨询
- 源材料丰富但范围明确（27 本麻醉学 PDF）
- 输出格式确定（PPTX，非交互式/非印刷）
- 时间充裕（可以迭代 v1→v9，没有硬性截止日期）

**警示：** 这些优势在其他项目中未必存在。移植方法论时，需要评估每个前提条件是否成立。
